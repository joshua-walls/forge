Forge includes two direct cleanup commands for metadata consistency:

```text
Forge: Normalize Tags
Forge: Normalize Frontmatter
```

These commands modify notes directly.

Run Vault Lint first so you understand what problems you are trying to fix before applying broad normalization.

# Normalize Tags

Tag normalization helps reduce long-term metadata drift.

Run:

```text
Forge: Normalize Tags
```

Forge can:

- trim empty tag values
- deduplicate tags case-insensitively
- sort tags predictably
- convert legacy `namespace:value` tags into `namespace/value`
- remove invalid structural tags used incorrectly as normal tags

Examples of structural namespaces include:

- `domain`
- `status`
- `type`

These are usually better represented as frontmatter fields instead of tags.

---

# Example Tag Cleanup

Example input:

```yaml
tags:
  - Topic:Forge
  - topic/forge
  - status/active
```

may normalize into:

```yaml
tags:
  - Topic/Forge
```

This helps reduce:

- duplicate tags
- inconsistent casing
- namespace drift
- query inconsistencies

# Why Tag Consistency Matters

Tag drift accumulates surprisingly quickly in long-lived vaults.

Examples:

- `project/active`
- `Project/Active`
- `project:active`
- `projects/active`

All of these may represent the same idea while fragmenting queries and dashboards.

Normalization helps keep tags predictable for:

- Dataview
- Bases
- dashboards
- exports
- automation workflows
- AI-assisted retrieval

---

# Normalize Frontmatter

Frontmatter normalization reorders fields into a consistent structure.

Run:

```text
Forge: Normalize Frontmatter
```

Forge uses the configured frontmatter field order when rewriting notes.

Example recommended order:

```text
type
status
tags
created
updated
ai_private
review_cycle
```

Fields not explicitly listed are appended alphabetically afterward.

---

# Why Frontmatter Ordering Matters

Consistent field ordering improves:

- readability
- diff clarity
- review workflows
- template consistency
- schema adoption
- long-term maintenance

Without normalization, metadata order gradually becomes inconsistent across the vault.

That becomes harder to review as vault size grows.

---

# What Normalization Does Not Do

Normalization is intentionally conservative.

It does not:

- rewrite prose
- reorganize note bodies
- repair Shapes
- infer missing metadata
- modify heading structure

It focuses only on metadata cleanup and consistency.

---

# Recommended Workflow

A common workflow looks like this:

1. Run Vault Lint
2. Review findings
3. Normalize tags
4. Normalize frontmatter
5. Run Vault Repair if needed
6. Re-run Vault Lint

This gradual cleanup cycle helps reduce metadata drift over time.

---

# When To Use These Commands

Normalization is especially useful after:

- importing notes
- changing schemas
- applying large patches
- migrating vault structures
- consolidating workflows
- cleaning older vault areas

Most users should begin with smaller folder scopes before normalizing the entire vault.

---

# High-Risk Areas

For sensitive or operational folders:

- use Git if possible
- test against small scopes first
- prefer dry-run patch workflows initially
- review normalization results carefully

Small controlled cleanup passes are usually safer than giant vault-wide rewrites.

---

# Relationship to Schemas and Lint

Schemas define the expected metadata structure.

Vault Lint detects drift from that structure.

Normalization helps bring metadata back toward consistency.

Together they help keep vaults:

- easier to query
- easier to maintain
- more reliable for Dataview
- more reliable for Bases
- easier to export
- easier to use with AI workflows
