Forge reads its schema from a markdown note, usually:

```text
{{schemaFile}}
```

The schema defines the structure Forge expects across the vault.

Linting, exports, repairs, normalization, dashboards, Bases, Dataview queries, and relationship indexes all become more reliable when this structure stays consistent.

You do not need a massive schema to get value from Forge.

Even a small schema can dramatically improve long-term vault consistency.

# Required Note Structure

A Forge schema is a normal markdown note with:

- frontmatter
- a `version::` inline field
- a `# Contract` heading
- one fenced YAML block under that heading

Example:

````md
---
type: reference
status: active
tags:
  - tool/forge
created: {{today}}
updated: {{today}}
ai_private: false
review_cycle: never
---

version:: "1.0"

# Contract

```yaml
meta:
  author: vault
  schemaRef: "{{schemaFile}}"

required_fields: []
optional_fields: []
inline_fields: []
tag_rules:
  require_namespace: true
  unknown_tags: warning
  severity: warning
  allowed_namespaces: []
exempt_paths: []
```
````

Forge reads the schema version from:

```text
version:: "1.0"
```

Do not place `meta.version` inside the YAML contract.

---

# What Schemas Do

Schemas help Forge:

- validate metadata
- detect structural drift
- standardize note types
- improve Dataview consistency
- improve Bases reliability
- normalize metadata
- build cleaner exports
- validate Shapes
- support AI-assisted workflows

Without a schema, metadata gradually drifts apart over time.

Examples:

- `project` vs `projects`
- `active` vs `in-progress`
- `type` vs `note_type`
- malformed dates
- inconsistent tags

Schemas help keep those patterns predictable.

---

# Field Definitions

A field definition can include:

| Key | Meaning |
|---|---|
| `name` | Frontmatter field name |
| `type` | `enum`, `string`, `boolean`, `date`, `list`, or `version` |
| `values` | Allowed values for `enum` fields |
| `severity` | `error`, `warning`, or `info` |
| `min_items` | Minimum array length for list fields |
| `strict_parse` | Date parsing flag used during validation |
| `stale_after_days` | Warn or error when a date becomes stale |
| `description` | Human-readable explanation |
| `lint_rules` | Conditional or consistency rules |

---

# Required vs Optional Fields

`required_fields` are checked on every non-exempt markdown note.

Use these for metadata that should exist consistently across the vault.

Examples:

- `type`
- `status`
- `created`

`optional_fields` are validated only when present.

Use optional fields for metadata that:

- is useful
- should remain valid
- but is not required everywhere

Examples:

- `owner`
- `reviewed_by`
- `source`
- `client`

This allows gradual adoption instead of forcing strict metadata immediately.

---

# Supported Field Types

| Type | Valid value |
|---|---|
| `enum` | A string matching one of `values` |
| `string` | A YAML string |
| `boolean` | `true` or `false` |
| `date` | `yyyy-MM-dd` |
| `list` | A YAML array |
| `version` | A string or number |

---

# Enum Example

Example:

```yaml
required_fields:
  - name: status
    type: enum
    values:
      - draft
      - active
      - archived
```

Forge validates that `status` only contains allowed values.

This helps keep dashboards, Dataviews, and Bases predictable.

---

# Conditional Lint Rules

Fields can include conditional rules using `lint_rules`.

Example:

```yaml
required_fields:
  - name: project
    type: string
    severity: warning
    lint_rules:
      - rule: required_when
        field: type
        equals: [project]
        severity: error
```

In this example:

- `project` becomes required
- only when `type` equals `project`

This allows schemas to model more realistic workflows.

---

# Supported Conditional Rules

| Rule | Meaning |
|---|---|
| `required_when` | Field must exist when another field matches a value |
| `forbidden_when` | Field must not exist when another field matches a value |
| `tag_consistency` | Field value should map to a matching tag namespace |

Conditional rules help keep metadata relationships consistent across the vault.

---

# Tag Rules

Forge can validate tag formatting and namespaces.

Example:

```yaml
tag_rules:
  require_namespace: true
  unknown_tags: warning
  severity: warning
  allowed_namespaces:
    - meta
    - source
    - tool
    - topic
```

When namespace rules are enabled:

```text
topic/forge
```

is preferred over:

```text
forge
```

This helps prevent uncontrolled tag drift in larger vaults.

---

# Inline Metadata

`inline_fields` lists body metadata keys allowed in:

```text
key:: value
```

format.

Example:

```yaml
inline_fields:
  - source
  - version
  - schema_version
```

Forge can detect:

- undocumented inline keys
- typo-like keys
- collisions with frontmatter fields

This helps keep inline metadata predictable.

---

# Exempt Paths

Use `exempt_paths` for folders Forge should ignore during linting.

Example:

```yaml
exempt_paths:
  - "{{forge}}"
  - "Templates"
  - "Archive"
```

Common exclusions include:

- template folders
- archives
- generated files
- scratch workspaces
- imported content

Forge also automatically excludes the Forge system folder from runtime lint processing.

---

# Schema Validation

After editing the schema, run:

```text
Forge: Validate Schema
```

Forge validates:

- YAML structure
- field definitions
- enum rules
- malformed configuration
- duplicate definitions
- invalid rule structure

Fix schema issues before running Vault Lint.

# Recommended Starting Point

Start small.

You do not need a giant schema immediately.

A simple schema with:

- `type`
- `status`
- tags
- basic date fields

is enough to begin improving consistency across the vault.

You can expand the schema gradually over time.
